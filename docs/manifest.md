# Прогнозирование спроса для оптимизации складских запасов
# 1. Проблематика
Интернет-магазин регулярно сталкивается с двумя крайностями: популярные товары быстро заканчиваются на складе, мы теряем чистую прибыль и лояльность клиентов, а непопулярные так и остаются на складе, замораживая оборотный капитал. Сейчас закупки планируются примерно или по простейшим правилам в Excel, что приводит к кассовым разрывам и хаосу в логистике.
# 2. Ценность для бизнеса
Внедрим систему автоматического прогнозирования спроса. Это позволит удерживать дефицит ходовых позиций в пределах 5% и одновременно сократить объем простаивающих излишков на складе на 20%.
# 3. Задачи проекта
- Разработать ML-модель для точного предсказания спроса на товары на 1 день вперед.
- Построить автоматический конвейер для регулярного переобучения на свежих данных.
- Реализовать механизм автоматического вывода модели из эксплуатации (rollback/сине-зеленый деплой), если ее метрики качества упадут ниже допустимого порога.
- Развернуть легковесный API-сервис для интеграции прогнозов с существующей системой управления заказами.
# 4. Архитектура
Признаки: история продаж за прошлые периоды (лаги: 1, 7, 14 дней), день недели, календарь праздников, текущая цена товара и остаток на складе.  
Модель: Градиентный бустинг LightGBM (задача регрессии).  
Инфраструктура: CI/CD через GitHub Actions, фичи хранятся как CSV-файлы в облачном хранилище (или пересчитываются из сырых данных при обучении), веб-сервис на FastAPI. Метрики собираются скриптом мониторинга; при превышении порога система автоматически инициирует замену модели или отправляет уведомление разработчику (Telegram/лог).  
Рамки и допущения: Прогнозы рассчитываются пакетно (batch) один раз в сутки. Система выдает только рекомендацию по объемам, сам процесс закупки товаров у поставщиков остается ручным.  

Уровень зрелости: 1
# 5. Ресурсы и осуществимость
Проект полностью реализуем за 7 рабочих дней силами команды из 2 человек. Весь стек технологий бесплатный и доступен без ограничений: Python, Docker, FastAPI, LightGBM. Для развертывания сервиса используем облачную платформу с бесплатным тарифом (Render / Railway / Amvera Cloud).
# 6. Стратегия работы с данными
Источник данных для демо: три CSV-файла (товары, склад, заказы), лежащие в репозитории. В продакшене данные выгружаются из облачной транзакционной БД.  
Целевая переменная: реальный объем продаж конкретного товара за день (задача регрессии).
# 7. Ключевые метрики
Бизнес-метрика: Fill Rate >= 95% (процент запросов клиентов, который склад смог удовлетворить сразу).
ML-метрики: MAPE <= 20% (средняя абсолютная ошибка в процентах), дополнительно отслеживаем RMSE для контроля крупных промахов.
# 8. Валидация и контроль качества
Офлайн (разработка): Кросс-валидация на временных рядах (Time Series Split) и финальная проверка на отложенной выборке за последний месяц.  
Онлайн (продакшн): Каждый день считаем MAPE, сравнивая вчерашний прогноз с реальными продажами. Порог деградации - 30% на скользящем окне в 7 дней. При превышении порога автоматически запускается пайплайн переобучения, и новая модель через CI/CD заменяет старую (сине-зеленый деплой или обновление переменной окружения).
# 9. Эволюция модели
Начнем с простой линейной регрессии в качестве базового решения (Baseline). Затем протестируем Random Forest и перейдем к LightGBM с тонкой настройкой гиперпараметров через Optuna для  максимального качества.
# 10. Формат инференса
Пакетный(Batch). Пайплайн запускается автоматически каждую ночь в 02:00, рассчитывает прогнозы на день вперед и сохраняет их в базу данных. Сервис FastAPI мгновенно отдает уже готовые, предрассчитанные значения по запросу системы заказов.
# 11. Обратная связь
Система непрерывно собирает логи прогнозов и реальные факты продаж. Реализован полноценный механизм Model Driven Deployment (MDD), если скользящий MAPE текущей модели пробивает порог в 30%, трафик автоматически переключается на резервную или свежепереобученную модель, которая показала лучшие результаты на валидации через перевыпуск контейнера или подмену версии в API-сервисе.
