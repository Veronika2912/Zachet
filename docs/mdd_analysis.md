# MDD-анализ: принятие решений на основе метрик (Metrics-Driven Development)

# Введение

MDD (Metrics-Driven Development) — подход, при котором архитектурные и продуктовые решения принимаются на основе количественных метрик работы системы. В данном анализе мы используем предоставленные данные о времени отклика (latency) для обоснования выбора между существующей и улучшенной версией API-сервиса.

# Входные данные

Для анализа предоставлены два распределения времени отклика системы:

| Система | Распределение | Среднее latency | Стандартное отклонение |
|---------|---------------|-----------------|----------------------|
| Существующая (базовая) | N(3.5, 0.4) | 3.5 сек | 0.4 сек |
| Улучшенная (новая) | N(2.0, 0.4) | 2.0 сек | 0.4 сек |

*Примечание: данные являются синтетическими и используются для демонстрации MDD-подхода. Реальные значения latency нашего сервиса (LightGBM + FastAPI) находятся в диапазоне 10-50 мс.*

# Визуализация

График показывает ядровые оценки плотности распределения (KDE):
- **Красная кривая** — существующая система (latency ~3.5 сек)
- **Зеленая кривая** — улучшенная система (latency ~2.0 сек)

Обе системы имеют одинаковую дисперсию (0.4 сек), что говорит о сохранении стабильности.

# Ключевые выводы

1. **Снижение среднего времени ответа:** с 3.5 сек до 2.0 сек (ускорение на 43%)
2. **Стабильность не ухудшилась:** дисперсия осталась неизменной (0.4 сек)
3. **Бизнес-эффект:** сокращение времени ожидания ответа API позволяет быстрее обрабатывать запросы системы заказов и снижает количество таймаутов

# Архитектурное решение на основе MDD

На основе анализа двух распределений принимаем решение в пользу **улучшенной системы** (зеленая кривая), так как она обеспечивает более низкую латентность при сохраненной стабильности.

### 1. Устанавливаем SLO для времени ответа

Для обеих систем задаем целевые показатели:

| Порог | Значение | Действие |
|-------|----------|----------|
| Target (цель) | p95 latency < 500 мс | Нормальная работа |
| Warning (предупреждение) | p95 latency > 400 мс в течение 10 минут | Уведомление в лог/Telegram |
| Critical (критический) | p95 latency > 500 мс в течение 5 минут | Переключение на резервную модель или rollback |

*Обоснование порогов:* 500 мс - консервативное значение с учётом возможных сетевых задержек и холодного старта в бесплатном облаке. Сам инференс модели LightGBM занимает ~10 мс.

### 2. Автоматические действия при нарушении SLO

| Условие | Действие |
|---------|----------|
| latency p95 > 500 мс (5 минут) | Переключить трафик на предыдущую стабильную версию модели (через /model/reload) |
| latency p95 > 400 мс (10 минут) | Отправить уведомление в Telegram/лог для инженера |
| Два нарушения за час | Заблокировать текущую версию модели, провести ручной разбор инцидента |

### 3. Механизм реализации в нашей системе

В нашем API-сервисе (FastAPI) реализован эндпоинт `/model/reload`, который:
- Загружает новую модель из `models/model.txt`
- Обновляет глобальную переменную `CURRENT_MODEL` в памяти
- Работает в фоновом потоке (BackgroundTasks), не блокируя обработку запросов

**Как работает автоматическое переключение при деградации качества:**

При превышении порога MAPE (30% на скользящем окне 7 дней) CI/CD пайплайн (GitHub Actions):
1. Обучает новую модель на свежих данных
2. Сохраняет артефакты в репозиторий (model.txt, metrics.json)
3. Вызывает `POST /model/reload` на прод-сервисе
4. Трафик автоматически переключается на новую модель без остановки сервера (Blue-Green Deployment)

# Почему это решение обосновано

| Критерий | Обоснование |
|----------|-------------|
| **Бизнес-ценность** | Более быстрое API = меньше таймаутов = более точное планирование закупок = выше Fill Rate |
| **Безопасность** | Возможность обратного переключения (rollback) защищает от деградации после обновления |
| **Автоматизация** | Исключает человеческий фактор при реагировании на инциденты (скорость реакции < 15 минут) |
| **Измеримость** | Все метрики логируются (metrics.json, логи API) и доступны для аудита и постмортема |

# Метрики успеха внедрения

| Метрика | Целевое значение | Способ измерения |
|---------|------------------|------------------|
| Время реакции на деградацию | < 15 минут | Время от превышения порога MAPE до вызова /model/reload |
| Доля автоматических переключений | > 90% | (Авто-переключения / все переключения) * 100% |
| Ложные срабатывания | < 5% | Доля переключений, после которых MAPE не улучшился |
| Доступность API | 99.5% | (Успешные ответы / всего запросов) * 100% |

# Заключение

На основе MDD-подхода (сравнение двух распределений latency) принимаем решение:

**Внедрить улучшенную систему с целевым SLO p95 latency < 500 мс и автоматическим переключением моделей при деградации качества.**

Это решение обеспечивает баланс между производительностью, надёжностью и автоматизацией, что соответствует уровню зрелости MLOps 1 и требованиям задания.
